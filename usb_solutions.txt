Here's what this means and why it's actually by design:

Why the error happens
libusb needs exclusive raw USB access to a device to forward it over USB/IP. But Windows won't allow that when another kernel driver already owns the device:

USBSTOR = Windows driver for USB mass storage (drives, dongles using storage class)
bthusb = Windows Bluetooth stack driver
HidUsb, usbvideo, etc. — same situation for other classes
Windows's driver model: only one kernel driver can own a USB device at a time. libusb loses this fight unless WinUSB is substituted.

What Zadig actually does
Zadig replaces the device's current kernel driver with WinUSB, a generic Microsoft driver that lets user-mode applications (like libusb) talk directly to the USB device. After that, libusb wins.

The trade-off — and why it's intentional for passthrough
When you replace a BT adapter's driver with WinUSB:

The local Windows BT stack (bthusb.sys) loses that adapter — your PC can no longer see local BT devices through it
But the passthrough forwards the raw USB device to the remote PC, where the remote BT stack takes over and owns it there
This is the whole point: the device is "detached" from the local PC and "given" to the remote PC. It's not a bug — it's the USB/IP model.

irtualHere solves this at a completely different level — and understanding it explains exactly why our approach requires a driver swap while theirs doesn't.

How VirtualHere works (the fundamental difference)
Our approach: user-mode (libusb)
libusb runs entirely in user space. Windows won't let a user-mode application open a device that an existing kernel driver has already claimed. Hence the need to replace the driver.

VirtualHere's approach: kernel-mode filter driver
VirtualHere installs vhusb64.sys as a WDM lower filter driver on the USB hub or device stack. It sits below the class driver in the kernel device stack. When you click "Use" on a device:

It sends a surprise removal IRP (IRP_MN_SURPRISE_REMOVAL) to the existing class driver — effectively tells USBSTOR/HidUsb/etc. "your device just vanished"
The class driver releases its handles and unloads from the device
VirtualHere's filter driver now exclusively owns the raw USB traffic at the kernel level
It redirects all USB URBs over the network to the server
No driver replacement, no Zadig, no UAC prompt — because it's already running in the kernel with the authority to do all of this.


Why we can't trivially do the same
VirtualHere	Our implementation
Driver level	Kernel (Ring 0)	User mode (Ring 3)
Driver signing	EV certificate + WHQL	Not required
Can eject class drivers	Yes, via IRP	No
Requires driver swap	Never	Once per device type
Complexity	Huge (kernel dev)	Manageable
UAC prompt	Never	Once (our auto-install)
Kernel drivers on modern Windows (since Win10 2016) require WHQL certification or an EV code-signing certificate + Microsoft attestation signing to load. This is a paid, weeks-long process — not feasible for a hobby project.

A middle ground that exists in our codebase
Our passthrough-server/ already uses usbip-win2, which IS a kernel-mode VHCI driver (server side). On the client side, usbip-win2 also has a stub driver (usbip_stub.sys) designed exactly to replace class drivers for forwarded devices. It's purpose-built for USB/IP — more appropriate than WinUSB for our use case but has the same limitation: you still need to swap the driver once (it just stays put for future sessions).

The auto-WinUSB install we just added is essentially doing what the usbip-win2 stub installer does, but integrated into Moonlight so the user never touches Zadig. One UAC prompt, one time per device type — then it's seamless forever after, including auto-forward on reconnect.
